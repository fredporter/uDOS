### Steps

1. **Phase 0: Initialization** ✅
   - Mission metadata (ID, name, version, timestamps)
   - Configuration variables (category, batch_size, quality_threshold, max_retries)
   - Paths (topics, knowledge base, drafts, state, checkpoints)
   - Statistics tracking (articles generated/approved/rejected/regenerated/committed)
   - API usage tracking (requests, tokens, cost)
   - Quality score accumulators (completeness, clarity, accuracy, usefulness)
   - Prerequisites verification (topics file, knowledge base directory)
   - Working directory creation
   - Initial checkpoint

2. **Phase 1: Gap Analysis** 🚧
   - Load knowledge_topics.json master list
   - Get existing guides from knowledge/{category}/
   - Compare current vs. planned topics
   - Calculate gap size and priority ranking
   - Generate gap report (JSON)
   - Save to memory/workflows/state/gaps_{category}.json
   - User selection of category (if multiple)
   - Checkpoint: gap_analysis_complete

3. **Phase 2: Content Generation** 🚧
   - Load gap report for selected category
   - Select batch of N topics (priority-ranked)
   - For each topic in batch:
     * Rate limiting check (API delay)
     * Execute: `GENERATE GUIDE "{title}" --category {cat} --style survival --output {path}`
     * Track generation time, word count
     * Update API usage (requests, tokens, cost estimate)
     * Save checkpoint after each article
   - Save batch statistics to state/batch_generation_{category}.json
   - Checkpoint: content_generation_complete

4. **Phase 3: Quality Review** 🚧
   - Load batch statistics
   - For each generated article:
     * Execute: `DOCS REVIEW {file}`
     * Extract quality scores (completeness, clarity, accuracy, usefulness, overall)
     * If score >= threshold: APPROVE
     * If score < threshold and retries available:
       - Extract improvement suggestions
       - Execute: `DOCS REGEN {file} --suggestions "{suggestions}"`
       - Re-review regenerated article
       - If new score >= threshold: APPROVE (after regen)
       - Else: FAIL (manual review needed)
     * Track approved/failed/regenerated counts
   - Calculate approval rate and average quality scores
   - Save review results to state/review_results_{category}.json
   - Checkpoint: quality_review_complete

5. **Phase 4: Standards Validation** 🚧
   - Load review results
   - For each approved article:
     * Markdown structure validation:
       - Heading hierarchy check
       - Code blocks properly fenced
     * Content standards:
       - Safety warnings (if applicable)
       - Step-by-step procedures (if applicable)
     * Metadata validation:
       - Category present
       - Tags present
   - Record validation pass/fail with error details
   - Save validation results to state/validation_results_{category}.json
   - Checkpoint: validation_complete

6. **Phase 5: Smart Commit** 🚧
   - Load validation results
   - For each validated article:
     * Prompt if file exists in knowledge/ (overwrite protection)
     * Move from drafts/ to knowledge/{category}/
     * Execute: `git add {file}`
   - Generate batch commit message with article list
   - Execute: `git commit -m "{message}"`
   - Save commit log to state/commit_log_{category}.json
   - Update knowledge bank index (optional)
   - Checkpoint: commit_complete

7. **Phase 6: Mission Reporting** 🚧
   - Calculate mission duration
   - Load all phase results (gap, batch, review, validation, commit)
   - Calculate coverage metrics (before/after counts, increase %)
   - Generate mission report JSON:
     * Mission metadata (ID, name, category, timestamps)
     * Gap analysis (gaps identified)
     * Generation (articles generated, failed)
     * Review (approved, failed, regenerated, approval rate)
     * Quality (average scores by dimension)
     * Validation (passed, failed)
     * Commit (articles committed)
     * Coverage (before, after, increase %)
     * API usage (requests, tokens, cost)
     * Errors encountered
     * Success flag (>= 80% committed)
   - Save to state/mission_report_{mission_id}.json
   - Display summary
   - Archive workflow state
   - Checkpoint: mission_complete

## Architecture Decisions

### 1. Data Flow Design

```
knowledge_topics.json (master list)
  ↓
Gap Analysis (compare with knowledge/{category}/*.md)
  ↓
gap_report.json (priority-ranked topics)
  ↓
Content Generation (GENERATE GUIDE batch)
  ↓
batch_generation.json (generated articles + stats)
  ↓
Quality Review (DOCS REVIEW + REGEN loop)
  ↓
review_results.json (approval/scores)
  ↓
Standards Validation (markdown/content/metadata checks)
  ↓
validation_results.json (pass/fail)
  ↓
Smart Commit (move to knowledge/, git commit)
  ↓
commit_log.json (committed files)
  ↓
Mission Reporting (aggregate all phase data)
  ↓
mission_report.json (final statistics)
```

### 2. State Management Pattern

**Location:** `memory/workflows/state/`

**Files:**
- `gaps_{category}.json` - Gap analysis results
- `batch_generation_{category}.json` - Generation statistics
- `review_results_{category}.json` - Quality review outcomes
- `validation_results_{category}.json` - Validation pass/fail
- `commit_log_{category}.json` - Committed files log
- `mission_report_{mission_id}.json` - Final report

**Rationale:** Each phase saves its results to a JSON file. Subsequent phases load previous results. Enables:
- Resumability (re-run failed phases)
- Debugging (inspect intermediate state)
- Auditing (full mission history)
- Reporting (aggregate phase data)

### 3. Checkpoint System

**Location:** `memory/workflows/checkpoints/`

**Checkpoints:**
1. `init-complete` - Initialization finished
2. `gap-analysis-complete` - Gap report generated
3. `content-generation-complete` - Batch generated
4. `quality-review-complete` - Review finished
5. `validation-complete` - Validation finished
6. `commit-complete` - Articles committed
7. `mission-complete` - Final report generated

**Rationale:** Save workflow state after each major phase. If workflow crashes/interrupted, can resume from last checkpoint. Each checkpoint includes:
- Timestamp
- Phase name
- Critical variables (mission ID, category, counts)
- Phase-specific state (topic list, current topic, etc.)

### 4. Integration Points

#### GENERATE GUIDE Integration (v1.1.15+)

**Command:** `GENERATE GUIDE "{title}" --category {category} --style survival --output {file}`

**Example:**
```upy
EXEC "GENERATE GUIDE 'Water Testing Methods' --category water --style survival --output memory/drafts/knowledge/water/water_testing.md"
```

**Returns:** Success/failure, word count, generation time

**Rate Limiting:**
- Check API.LAST_REQUEST_TIME
- Wait CONFIG.API_DELAY seconds if needed
- Update API.TOTAL_REQUESTS, API.TOTAL_TOKENS, API.ESTIMATED_COST

#### DOCS REVIEW Integration (v1.1.17+)

**Command:** `DOCS REVIEW {file}`

**Example:**
```upy
EXEC review_result = "DOCS REVIEW memory/drafts/knowledge/water/water_testing.md"
```

**Returns:**
- `score.overall` (0-100)
- `score.completeness` (0-100)
- `score.clarity` (0-100)
- `score.accuracy` (0-100)
- `score.usefulness` (0-100)
- `issues` (list of problems)
- `suggestions` (improvement recommendations)

#### DOCS REGEN Integration (v1.1.17+)

**Command:** `DOCS REGEN {file} --suggestions "{suggestions}"`

**Example:**
```upy
EXEC regen_result = "DOCS REGEN memory/drafts/knowledge/water/water_testing.md --suggestions 'Add safety warnings, expand step-by-step instructions'"
```

**Returns:** Success/failure, new word count

**Logic:**
- Only call if review score < threshold
- Only retry MAX_RETRIES times
- Re-review after regeneration
- Track as "approved_after_regen" or "failed_review"

## Challenges & Solutions

### Challenge 1: Variable Namespace Design

**Problem:** Need consistent variable naming across 6 phases. Should we use `MISSION.*`, `WORKFLOW.*`, `API.*`, `STATS.*`, `PATH.*`?

**Solution:** Use namespace prefixes for clarity:
- `MISSION.*` - Mission-level config and state (category, batch_size, status, timestamps)
- `WORKFLOW.*` - Workflow execution state (phase, iteration, current_topic, errors)
- `API.*` - API usage tracking (requests, tokens, cost, last_request_time)
- `STATS.*` - Accumulated statistics (quality scores, counts)
- `PATH.*` - File paths (topics, knowledge_base, drafts, state, checkpoints)

**Rationale:** Clear separation of concerns, easy to grep, self-documenting.

### Challenge 2: Checkpoint Granularity

**Problem:** How often to checkpoint? After each article (expensive), or after each phase (coarse)?

**Solution:** Hybrid approach:
- Major checkpoints after each phase (6 total)
- Minor checkpoints after each article generation (resumability)
- Store minimal state (topic ID, counts) in minor checkpoints
- Full state snapshot in major checkpoints

**Rationale:** Balance resumability with storage overhead. Generation is the longest phase (10+ articles), so checkpoint each article. Other phases are fast, so one checkpoint per phase is sufficient.

### Challenge 3: Error Handling Strategy

**Problem:** What if GENERATE fails? What if DOCS REVIEW unavailable? Should workflow abort or continue?

**Solution:** Graceful degradation:
- GENERATE failure: Log error, skip topic, continue batch
- DOCS REVIEW failure: Log warning, auto-approve (or manual review flag)
- Git commit failure: Save commit log, prompt manual commit
- Track errors in WORKFLOW.ERRORS array
- Set MISSION.SUCCESS based on 80% threshold (8/10 articles committed)

**Rationale:** Don't abort entire workflow for single failures. Continue batch, report errors, require 80% success rate.

## Lessons Learned

1. **Start with Structure**: Defining the 6-phase architecture first (before implementation) clarified data flow, state management, and integration points.

2. **Variable Namespaces**: Using prefixes (`MISSION.*`, `WORKFLOW.*`, `API.*`) makes large workflows more maintainable and self-documenting.

3. **Incremental Testing**: Plan to test each phase individually before end-to-end testing. Reduces debugging complexity.

4. **State Persistence**: Saving phase results to JSON files enables debugging, resumability, and auditing. Worth the extra I/O.

5. **Checkpoint Strategy**: Hybrid (major + minor) checkpoints balance resumability with storage overhead.
