# uDOS Versioning Philosophy

## Current Reality: We Are Building v1.0

uDOS is a foundational system still in active development toward its first stable release. The version progression during development has helped us move forward, but we need to properly reflect our actual maturity.

### Proposed Version Reset
- **Current Development**: v1.0.4.1 (building toward v1.0 stable)
- **Remove version numbers from filenames** - docs should be living documents
- **Focus on foundational capabilities** rather than "enhanced" or "latest" features

## Documentation Philosophy Changes

### Language Principles
✅ **Use**: foundational, core, essential, clean, simple, lean, fast
❌ **Avoid**: latest, updated, enhanced, extended, legacy (in dev context)

### "Legacy" Term Clarification
- **Legacy in uDOS workflow**: Move → Milestone → Mission → **Legacy** (positive endpoint)
- **Legacy = Worth Keeping**: What each installation strives to leave for the next user
- **Not for dev docs**: Avoid "legacy code" or "legacy systems" terminology in development

### Version-Free Documentation
- Remove version numbers from filenames
- Write docs as living documents that evolve
- Create development snapshots in `/dev/` folder when needed
- Focus on building solid foundations, not feature extensions

## Implementation Plan

1. **Reset versioning** to v1.0.4.1 across system
2. **Update documentation language** to reflect foundational work
3. **Remove version numbers** from doc filenames
4. **Create dev snapshots** in `/dev/docs/` for historical reference
5. **Refocus messaging** on core capabilities and simplicity

This approach better reflects our actual development stage and avoids confusion about maturity level.
