#### [docs/specs/uCLI-COMMAND-DISPATCH-v1.4.4.md](../../docs/specs/uCLI-COMMAND-DISPATCH-v1.4.4.md)

Three-stage input dispatch chain specification:
- **Stage 1 (uCODE):** Exact/alias/abbreviation matching against canonical command registry
- **Stage 2 (Shell):** Validate shell syntax safety; execute if safe (bash passthrough)
- **Stage 3 (VIBE/OK):** Route to Wizard AI service for natural language handling
- Confidence scoring and user prompts for ambiguous inputs
- Safety checks (prevent injection attacks, dangerous commands)
- Debug mode (`--dispatch-debug`) for routing visibility
- Performance targets (<10ms uCODE, <50ms shell validation, <500ms VIBE lookup)

**Artifacts:** Dispatch service implementation, shell validator, VIBE routing, comprehensive test suite

## Architecture & Standards (Updated)

### Code Quality Standards (v1.4.4 - Expanded)

From error-handling spec:
- All errors use structured `CommandError` with code + message + recovery_hint
- 100% of user-facing errors have recovery guidance
- No sensitive data in logs

From dispatch spec:
- Three-stage dispatch chain with well-defined interfaces
- Stage 1 confidence scoring for command matching
- Stage 2 safety checks prevent dangerous commands
- Stage 3 graceful fallback when Wizard unavailable

From lens spec:
- All game variants use isolated lens adapters
- SimpleAsciiLens fallback works for all variants
- Lens interface contract enforced via Protocol

### Testing Standards (v1.4.4 - Expanded)

- 90%+ coverage on command handlers (including new gameplay tests)
- 85%+ coverage on lens implementations
- 100% safety validation for shell commands
- Dispatch performance targets: <600ms total latency
- Gameplay adapter lifecycle tests mandatory

### Architectural Boundaries (New)

**TUI GENRE vs Wizard GUI Themes:**
- `/themes/genre/` — TUI styling (ANSI colors, ASCII art, error formats)
- `/wizard/web/themes/` — Web GUI styling (HTML/CSS, responsive design)
- No overlap; separate configuration namespaces

**Gameplay Lenses:**
- Each TOYBOX variant has dedicated lens module in `/core/lenses/`
- Lens interface abstracts game output → uDOS viewport
- External game runtimes never modified (adapters only)

**Command Dispatch:**
- Stage 1 (uCODE) — Core-only, no dependencies
- Stage 2 (Shell) — Isolated validation, subprocess safety checks
- Stage 3 (VIBE) — Optional, gracefully fails if Wizard offline

- All errors use structured `CommandError` with code + message + recovery_hint
- User errors (4xx) log at INFO level
- System errors (5xx) log at ERROR level
- 100% of user-facing errors have recovery guidance
- No sensitive data in ERROR logs

### Testing Standards (v1.4.4)

From modularization audit spec:
- No circular dependencies (acyclic DAG required)
- Core Python: stdlib-only, venv-independent
- Core TypeScript: no Node.js-only dependencies
- 90%+ coverage on command handlers
- 85%+ coverage on services
- 95%+ coverage on parsers

### Module Organization (v1.4.4)

- Functions >150 lines extracted to `*_helpers.py`
- Public API documented in docstrings + `__all__`
- Private functions prefixed with `_`
- Phase-specific logic extracted into separate handlers
- Services respect single-responsibility principle
